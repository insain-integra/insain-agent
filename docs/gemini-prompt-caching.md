# Кэширование промптов Gemini (Context Caching)

## Что это

**Context Caching** в Gemini позволяет один раз загрузить в кэш статичный контекст (system instruction, инструменты, длинные справочники) и в последующих запросах передавать только **идентификатор кэша** плюс новые сообщения. Закэшированные входные токены тарифицируются **дешевле**, чем при каждой отправке заново.

Поддерживается в том числе моделью **Gemini 3.1 Pro Preview** (и другими; минимальный объём для кэша — обычно от 1024 до 4096 токенов в зависимости от модели).

## Два режима

| Режим | Описание |
|-------|----------|
| **Неявное (implicit)** | Включено по умолчанию для многих моделей. Один и тот же префикс контекста может кэшироваться на стороне Google автоматически. Экономия не гарантирована, код менять не нужно. |
| **Явное (explicit)** | Вы создаёте кэш через API (`caches.create`), получаете имя кэша, в каждом запросе передаёте `cached_content=<имя>`. Гарантированная экономия при повторных запросах. |

## Что у нас можно кэшировать

В боте при каждом запросе к LLM отправляется:

- **Системный промпт** — введение, список калькуляторов, правила выбора материалов, форматирование (несколько тысяч токенов).
- **Список tools** — схемы калькуляторов и enum материалов (тоже большой объём).
- **История + новое сообщение** — переменная часть, её не кэшируем.

Идеальный кандидат для кэша: system instruction + tools. Они одинаковы для многих запросов подряд (пока не обновили список калькуляторов/опций).

## Как использовать в проекте

### Вариант 1: Неявное кэширование (без изменений кода)

1. В `.env` задать модель с поддержкой кэширования, например:
   ```env
   GEMINI_MODEL=gemini-2.5-flash
   ```
   или, когда будет доступна:
   ```env
   GEMINI_MODEL=gemini-3.1-pro-preview
   ```
2. Ничего не менять в коде. Google может автоматически кэшировать повторяющийся префикс запросов — экономия возможна, но не гарантирована.

### Вариант 2: Явное кэширование (требует доработки)

Явное кэширование в Gemini делается через **нативный API** (REST `cachedContents` или SDK `google-genai`), а не через OpenAI-совместимый эндпоинт. Текущий код бота использует `openai` + `base_url` на `generativelanguage.googleapis.com/v1beta/openai/` — параметр `cached_content` там, как правило, **не поддерживается**.

Чтобы использовать явный кэш:

1. **Добавить зависимость**  
   `google-genai` (официальный SDK для Gemini).

2. **При старте бота (или по таймеру)**  
   - Собрать system instruction + tools так же, как сейчас в агенте.  
   - Вызвать `client.caches.create(model=..., config=CreateCachedContentConfig(system_instruction=..., tools=..., ttl="3600s"))`.  
   - Сохранить возвращённое `cache.name`.

3. **При каждом запросе к LLM**  
   - Вызывать Gemini через нативный SDK:  
     `client.models.generate_content(..., config=GenerateContentConfig(cached_content=cache.name))`  
   - В контент передавать только историю диалога + новое сообщение пользователя.

4. **Учесть**  
   - При переключении на нативный SDK нужно будет дублировать или вынести логику выбора модели, fallback на YandexGPT и т.д.  
   - Кэш имеет TTL (например, 1 час); по истечении нужно создавать кэш заново.  
   - Для двухшаговой схемы (сначала классификатор калькулятора, потом расчёт) можно сделать отдельный кэш для шага расчёта: system prompt по конкретному калькулятору + один tool.

## Резюме

| Цель | Действие |
|------|----------|
| Попробовать без изменений кода | Выставить `GEMINI_MODEL=gemini-2.5-flash` (или `gemini-3.1-pro-preview`) и оставить текущую интеграцию — возможно получите неявное кэширование. |
| Гарантированная экономия | Реализовать явное кэширование через `google-genai`: создание кэша при старте/обновлении опций, использование `cached_content` в запросах (с адаптацией под наш двухшаговый агент и fallback на YandexGPT). |
